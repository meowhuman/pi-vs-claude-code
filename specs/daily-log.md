# Specification: Daily Log Extension

## 1. Overview

**Daily Log** is a lightweight work journaling extension for the Pi Coding Agent. It lets developers record daily progress, blockers, and decisions without leaving the Pi session. Each day's entries are persisted as a structured Markdown file under `logs/YYYY-MM-DD.md`.

**Usage:** `pi -e extensions/daily-log.ts`

---

## 2. Tools

### 2.1 `log_entry`

Add a new entry to today's log.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `section` | `"done" \| "blocked" \| "note" \| "decision"` | Yes | Which section to append the entry to |
| `text` | `string` | Yes | Entry content (plain text, supports inline code) |
| `tags` | `string[]` | No | Optional labels for filtering (e.g. `["backend", "auth"]`) |

**Behaviour:**
- Appends to today's file (`logs/YYYY-MM-DD.md`), creating it from template if absent.
- Returns confirmation with the rendered entry line.

---

### 2.2 `log_show`

Display today's log (or a specific date) in the Pi UI overlay.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `date` | `string` | No | ISO date `YYYY-MM-DD`. Defaults to today. |
| `section` | `"done" \| "blocked" \| "note" \| "decision" \| "all"` | No | Filter by section. Defaults to `"all"`. |

**Behaviour:**
- Reads the target file and renders it in a `DynamicBorder` overlay.
- Shows "No log for this date." if the file does not exist.

---

### 2.3 `log_summary`

Ask the agent to write a one-paragraph summary of the day's activity and append it to the `## Summary` section.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `date` | `string` | No | ISO date `YYYY-MM-DD`. Defaults to today. |

**Behaviour:**
- Reads all sections, generates a summary via the active model, appends under `## Summary`.
- Idempotent: replaces any existing summary rather than duplicating it.

---

### 2.4 `log_search`

Search across all log files for entries matching a keyword or tag.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `query` | `string` | Yes | Keyword or `#tag` to search for |
| `limit` | `number` | No | Max results to return. Defaults to `20`. |

**Behaviour:**
- Scans `logs/*.md` files, returns matching lines with file date as context.
- Results displayed in overlay, newest-first.

---

## 3. UX Interaction Flow

### 3.1 Session Start

On `session_start`, the extension:
1. Checks if today's log file exists.
2. If absent, silently creates it from template (no prompt required).
3. Sets the status line: `📓 Daily Log — 2026-03-31 | 0 entries`.

### 3.2 During Session

- Status line counter increments live as entries are added: `📓 Daily Log — 3 entries`.
- Agent can call tools freely; extension does not intercept or block any tool calls.
- `/daily-log` slash command opens the current day's overlay at any time.

### 3.3 Session End

On `session_end` (or when agent completes):
1. Display a non-blocking notification: "Daily log has N entries. Run `log_summary` to generate a summary."
2. No forced interaction — user dismisses or ignores.

### 3.4 Design Principles

- **Zero friction:** No required fields beyond `section` + `text`.
- **Non-intrusive:** Never blocks agent execution.
- **Glanceable:** Status line always shows today's count.
- **Recoverable:** Append-only writes; no entry is ever deleted by the extension.

## 4. Output Format — `logs/YYYY-MM-DD.md`

```markdown
# Daily Log — 2026-03-31

## Done
- Implemented JWT refresh logic in `auth.ts` #backend #auth
- Reviewed PR #42 — approved with comments

## Blocked
- Database migration stuck on foreign key constraint — waiting for DBA review #db

## Decisions
- Chose optimistic locking over pessimistic for the cart service (lower contention expected)

## Notes
- `bun test` is ~2× faster than previous `jest` setup
- TODO: revisit error boundary in `CheckoutPage`

## Summary

_(auto-generated by `log_summary`)_
Completed JWT refresh implementation and reviewed PR #42. The main blocker is a database migration
issue pending DBA input. Opted for optimistic locking in the cart service based on expected low
contention patterns.
```

### 4.1 File Rules

| Rule | Detail |
|------|--------|
| Location | `logs/YYYY-MM-DD.md` (relative to project root) |
| Encoding | UTF-8, Unix line endings |
| Created by | `log_entry` on first write of the day |
| Template sections | `Done`, `Blocked`, `Decisions`, `Notes`, `Summary` (all initially empty) |
| Append order | Newest entry last within each section |
| Tags | Inline `#tag` at end of entry line |

---

## 5. Extension Architecture Notes

- Register all tools at the **top level** of the extension function (not inside event handlers).
- Use `isToolCallEventType()` for type-safe tool call narrowing where applicable.
- File I/O via Node `fs/promises`; no external dependencies beyond `@mariozechner/pi-coding-agent`.
- Status line managed via `ctx.ui.setStatus()`.
- Overlay rendered via `DynamicBorder` (consistent with existing extensions).
- Today's date resolved once at `session_start` and stored in module scope.

---

## 6. Out of Scope (v1)

- Editing or deleting existing entries
- Export to external systems (Notion, Obsidian, etc.)
- Recurring templates per weekday
- Multi-project log separation
